**Purpose:** Wave 1–3 establish the unit-test contract. The Canvas runtime API is the only undocumented dependency in P16 — the unit tests prove the helpers handle the *expected* shape, but only manual E2E in real Obsidian proves the *actual* shape matches. This is the single highest-information test in the phase.

**Output:** `.planning/phases/16-canvas-reply-via-canvas-api/16-MANUAL-E2E.md` filled in with `[ ]`/`[x]` boxes, observed behaviors, console output snippets, and any deviations from expectation. If a scenario fails, this plan halts the phase and routes to `/gsd-plan-phase --gaps` for remediation.

**Why this can't be automated:** Per 16-VALIDATION.md, all five rows in §Manual-Only Verifications require either (a) the real Canvas runtime, (b) multi-leaf Obsidian state, (c) file-write race timing under real conditions, or (d) monkeypatching `Canvas.prototype` at runtime. None of these are reproducible in vitest without a fully working Canvas mock — which would itself be the bug the test is supposed to catch.

**Callout shape contract (per checker nit #8 reconciliation):** Plan 04 routes the canvas error UX through `deliverCanvasReply` with `buildErrorCallout(query, reason)` content. `buildErrorCallout` produces `> [!claude] {query}\n> {body}` (the `[!claude]` header for the error case — distinct from the success header `[!claude-done]+`). Scenario 4 below expects to see `> [!claude] {query}` in the canvas node + an error body line beneath it. This matches Plan 04's `writeCanvasErrorCallout` helper exactly.
</objective>

**Expected (per Plan 04's writeCanvasErrorCallout helper, which uses buildErrorCallout — checker nit #8 reconciliation):**

- An error callout appears in the canvas text node with the **`[!claude]`** header (not `[!claude-done]+` — that's the success header). Concretely:
  ```
  > [!claude] ping4
  > Canvas write failed: probe-failed
  ```
  The first line is `> [!claude] ping4` (the error header per `buildErrorCallout(query, reason)`), and a body line beneath it carries the failure reason.
- DevTools console shows a `console.error` line including the reason (`probe-failed` or similar) and any captured exception.
- Reload the plugin afterwards (the monkey-patch is sticky for the canvas instance).

**Why `[!claude]` and not `[!claude-done]+`:** `buildResponseCallout` (success) emits `[!claude-done]+`; `buildErrorCallout` (failure) emits `[!claude]`. Plan 04 routes the canvas error UX through `buildErrorCallout`, so the failure callout uses the error header — even though the write itself reaches the canvas via the same Canvas API pipeline. This is intentional and semantically correct.
